<task type="auto">
  <name>Task 1: Implement Status State Machine</name>
  <files>backend/internal/modules/blog/service.go</files>
  <action>
    Implement the business logic for status transitions (per D-01):
    - Allowed statuses: `Draft`, `Scheduled`, `Published`.
    - Validation: 
      - Transition to `Scheduled` requires a `published_at` date in the future.
      - Transition to `Published` requires `published_at` to be now or in the past.
    - Implement a `UpdatePostStatus` method in the Blog Service.
  </action>
  <verify>
    <automated>
      # Test invalid transition: Scheduled without date
      curl -X PATCH http://localhost:8080/api/v3/blog/posts/test-post/status -d '{"status": "Scheduled"}' | grep "published_at is required"
    </automated>
  </verify>
  <done>Status transitions are validated and enforced in the service layer</done>
</task>

<task type="auto">
  <name>Task 2: Implement Asynchronous Publishing Worker</name>
  <files>
    - backend/internal/modules/blog/worker.go
    - backend/internal/modules/blog/service.go
  </action>
  <action>
    Implement the automatic publishing mechanism (per D-02):
    1. Create a `PublishPostTask` payload for Asynq.
    2. In `UpdatePostStatus`, if status becomes `Scheduled`, enqueue a task with a delay until `published_at`.
    3. Implement the Asynq worker handler that calls the Blog Service to set the post status to `Published`.
    4. Ensure the worker handles cases where the post might have been deleted or changed back to `Draft` before the task ran.
  </action>
  <verify>
    <automated>
      # Schedule a post for 10 seconds from now, wait 15, check status
      curl -X PATCH http://localhost:8080/api/v3/blog/posts/test-post/status -d '{"status": "Scheduled", "published_at": "..."}' && \
      sleep 15 && \
      curl -X GET http://localhost:8080/api/v3/blog/posts/test-post | grep "Published"
    </automated>
  </verify>
  <done>Scheduled posts automatically transition to Published via Asynq</done>
</task>
